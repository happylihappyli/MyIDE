# AI C++ 乱码问题总结报告

## 1. 核心原因分析
之前多次尝试修复工具栏按钮中文乱码均失败，核心原因是**未解决源文件编码问题**。
在 Windows 环境下，G++ 默认使用 ANSI（如 GBK）解析源文件中的非 ASCII 字符。而我们的源文件通常是 UTF-8 编码（无 BOM），这导致宽字符字面量（如 `L"打开"`）在编译时被错误解析和编译，从而产生乱码。

## 2. 为什么其他 AI 的解决方案能生效？
1. **全面使用宽字符 API**：如 `CreateWindowExW`、`RegisterClassW`、`MessageBoxW` 等，避免了在程序内部发生 ANSI 到 Unicode 转换时的潜在问题。

## 3. 我们之前犯的错误及教训
- 盲目尝试在**运行时**进行转换（如使用 `MultiByteToWideChar`、`TB_ADDSTRING` 等），而没有从根本上解决**编译器解析源文件时**的编码错位问题。
- 混合使用了 ANSI 和 Unicode 版本的 API，导致数据传递时的混乱。
- 在提示词中，我们未主动说明源文件的编码格式，也未采用最稳妥的硬编码转义序列方案。

## 4. 未来的提示词改善建议
在后续要求 AI 编写或修改包含中文的 C++ Windows 代码时，应在提示词中包含以下信息：
- **明确环境**：说明源文件编码（如 UTF-8 without BOM）、构建系统（如 SCons）及编译环境（g++）。
- **统一 API**：明确要求**只使用 W 版本的 Win32 API**，确保内部数据流一致。

### 示例改善提示词：
> “工具栏按钮文本显示乱码。源文件保存为 UTF-8 without BOM，当前项目使用 SCons 和 g++ 编译。请修改代码，并全部使用 W 版本的 API，确保兼容性。”
